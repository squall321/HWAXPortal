## 슬라이드 1 — 표지 / 한 줄 비전

**HWAX : 개발실 데이터 & LLM Agent 통합 허브**

> "각 팀은 **자동화·데이터·Agent를 만드는 데**만 집중하고,
> **서비스화·연동·운영은 HWAX 허브가 표준으로 대행**한다."

- 부제: 만드는 건 팀, 운영은 허브 — 단 **3가지 약속**(공통 API · 컨테이너 규격 · GitHub 소스)을 지키면.

---

## 슬라이드 2 — Why (지금 추진하는 이유)

| Pain | 현재 | HWAX 이후 |
|---|---|---|
| **파편화** | 팀마다 stack·API·인증 제각각 → 연동/재사용 안 됨 | 공통 게이트웨이·SSO로 한 번에 연결 |
| **중복 투자/운영 낭비** | 팀마다 배포·인증·LLM연동을 따로 구축 | 허브가 대행 → 팀은 본업에 집중 |
| **AI/데이터 활용 정체** | 데이터·자동화가 흩어져 전사 LLM/Agent 활용 불가 | 허브로 모아 RAG·Agent 가속 |

> ※ 상용 LLM 사용은 **전사 차원 결정 완료** → 과금/보안 정책은 본 추진 범위 밖.

## 슬라이드 3 — 전체 구조 : 3-Tier 온보딩 사다리 (★구현 완료)

팀의 부담을 "어디까지 덜어주냐"의 3단계. **세 서비스 모두 현재 동작 중.**

```
                          ┌──────────────── HWAX Portal (단일 관문) ────────────────┐
                          │  SSO · 접근통제 · 타 그룹 포탈 링크(federation)          │
                          └──────────────────────────────────────────────────────────┘
   가벼움 ──────────────────────────────────────────────────────────────────► 무거움
   ┌─────────────────────┐   ┌─────────────────────────┐   ┌──────────────────────────┐
   │ Tier 1              │   │ Tier 2                  │   │ Tier 3                   │
   │ MX White Paper      │   │ HWAXHub                 │   │ AX Data Hub              │
   │ 링크/백서 카탈로그   │   │ 소스→빌드→서비스 대행    │   │ 데이터/RAG→LLM 대행       │
   │                     │   │ (내부 PaaS)             │   │                          │
   │ 팀: 포탈 URL 등록    │   │ 팀: GitHub 소스 제출     │   │ 팀: 데이터·Agent 등록     │
   │ 허브: 게시·검색      │   │ 허브: 빌드·배포·호스팅   │   │ 허브: 적재·임베딩·RAG·LLM │
   └─────────────────────┘   └─────────────────────────┘   └──────────────────────────┘
```

**근거(구현 실측)**:
- HWAXHub: GitHub 연동(`github_integration.py`) + **15개 런타임 SIF 템플릿**(C++/Python/Java/Go/Rust/.NET/Node/Next.js/R/Shiny/Streamlit/Dash…) + stack 자동판별 → **"느슨 표준"이 이미 구현**.
- AX Data Hub: `ingest`(적재)·`agents`·`groups`·`discover` 라우터 구현.
- Portal: `proxy`/`external-url`/`jwt-handoff`/`saml-handoff` 4종 연동 + sub-path 프록시(X-Forwarded-Prefix) 동작.

## 슬라이드 4 — R&R 경계 : Contract Line (★핵심)

```
        각 팀 (서비스 제공자)              │          HWAX 허브 (우리 팀 = 플랫폼)
─────────────────────────────────────────┼──────────────────────────────────────────
 ✅ 비즈니스 로직 / 도메인 기능            │ ✅ 빌드·배포 파이프라인 (CI/CD)
 ✅ 소스코드 (GitHub 제출)                 │ ✅ 호스팅·도메인·라우팅 (포탈 prefix)
 ✅ 데이터·문서·Agent 정의 콘텐츠          │ ✅ 공통 인증/SSO·게이트웨이
 ✅ 컨테이너 규격 준수 (느슨 표준)         │ ✅ RAG 인프라·LLM 게이트웨이
 ✅ API를 공통 규약(OpenAPI)으로 노출      │ ✅ 모니터링·로그·운영 자동화
 ✅ 1차 기능 테스트                        │ ✅ 표준·템플릿·온보딩 가이드 제공
─────────────────────────────────────────┴──────────────────────────────────────────
   Contract Line = 공통 API(OpenAPI+SSO) · 컨테이너 규격 · GitHub 소스 제출
   → 이 선을 팀이 넘겨주면, 그 아래(운영 전부)는 HWAX 허브가 책임진다.
```

**판정 룰** (누가 할지 헷갈릴 때):
- *"이게 이 서비스에만 고유한가?"* → **팀**
- *"이게 모든 서비스에 공통인가?"* → **허브**

## 슬라이드 5 — Top-down 요청 사항 (담당님 의사결정)

팀이 자발적으로 못 하는, **위에서 못 박아야** 굴러가는 것:

| # | 요청 항목 | 왜 Top-down |
|---|---|---|
| T1 | **HWAX를 "표준 경로"로 지정** ("새 자동화·서비스는 허브에 올린다"가 디폴트) | 표준화의 한 방 — 디폴트 아니면 아무도 안 옴 |
| T2 | **컨테이너 규격 + 공통 API/SSO 준수 의무화** (느슨 표준, 예외는 승인제) | 운영 대행이 성립하는 최소 조건 |
| T3 | **내부 GitHub 소스 공유 의무화** | 소스 없으면 빌드·서비스 자동화 불가 |
| T4 | **그룹별 "포탈/데이터 담당자(Owner)" 지정** ← *추가* | 아래 슬라이드 6 참조. 위에서 지정해야 책임 명확 |
| T5 | **R&R·리소스 주체 합의** (허브 운영=우리 팀 vs 팀 개발 리소스) | 예산·인력 경계 |

## 슬라이드 6 — 그룹별 담당자(Owner) 지정 (★추가 핵심)

> 그룹별 포탈이 있으면, 그 **포탈 시스템 관리 + 데이터 업로드 관리를 맡을 "LLM 능숙 담당자"를 그룹마다 1명 지정**.

| 담당자 역할 | 구체 업무 |
|---|---|
| **포탈 시스템 관리** | 그룹 포탈을 HWAX에 등록·갱신, 링크/접근 관리 |
| **데이터 업로드 관리** | 그룹 데이터·문서를 AX Data Hub에 적재, RAG 소스 큐레이션 |
| **Agent/LLM 운용** | 그룹 Agent 정의·튜닝, 질의 품질 관리 (LLM 능숙자) |
| **허브 창구** | 허브팀↔그룹 단일 접점 (온보딩·이슈) |

- **구현 연계**: AX Data Hub에 `groups` 개념은 있음 → **그룹별 `owner` 필드 + 권한**을 신설하면 이 R&R이 시스템에 박힘. (현재 owner 지정 필드는 미구현 → "신설 항목")
- **효과**: 데이터/Agent 품질이 "주인 있는 자산"이 됨 → 방치·정체 방지.

## 슬라이드 7 — HWAX 허브(우리 팀) 제공물

표준이 굴러가려면 우리가 만들어 줘야 하는 것 (= Contract Line 오른쪽 구체화):

| 영역 | 제공물 | 현황 |
|---|---|---|
| 공통 API | API 게이트웨이 · SSO 표준 · OpenAPI 템플릿 | SSO/프록시 동작, 게이트웨이 표준화 진행 |
| 컨테이너 규격 | 표준 def/이미지 + 15 런타임 템플릿 + 예외 승인 | **구현됨** (HWAXHub stacks) |
| CI/CD 자동화 | "GitHub push→빌드→서비스" 무인 파이프라인 | 부분 구현 (빌드·Drive 배포 자동화) |
| 데이터/RAG/LLM | 적재 표준 · RAG 파이프라인 · LLM 게이트웨이 | **구현됨** (AX Data Hub ingest/agents) |
| 운영 | 모니터링·로그·헬스·장애대응 | 부분 (진단/배포 자동화 보유) |
| 거버넌스 사무국 | 표준 문서·온보딩 가이드·예외 승인 창구 | 신설 |

> 강점: **플랫폼 코어(포탈·SSO·프록시·빌드·RAG)는 이미 깔려 있음** → 표준화는 "정착·확산" 단계.

## 슬라이드 8 — 각 팀 온보딩 체크리스트 (전파용)

성숙도에 맞는 칸을 고르면 됨:

**Tier 1 — 링크만 (제일 쉬움)**
- [ ] 팀 자동화 포탈 URL을 MX White Paper에 등록 → 끝

**Tier 2 — 서비스 대행 (HWAXHub)**
- [ ] 내부 GitHub 레포 제출 (소스 공유)
- [ ] `manifest`에 `build.stack` 선언 (지원 15종 중 택1, 언어 자유)
- [ ] 공통 API/SSO 규약 적용 (sub-path·인증)
- [ ] → push 하면 빌드·배포·호스팅 자동

**Tier 3 — 데이터/AI (AX Data Hub)**
- [ ] 데이터·문서 표준 포맷으로 등록
- [ ] RAG 소스·Agent 정의
- [ ] **그룹 담당자(Owner) 지정** (포탈/데이터 관리)
- [ ] → RAG·LLM 서비스 대행

## 슬라이드 9 — 추진 로드맵

```
Phase 0 (완료)  포탈 + 3허브 동작·연동                    ← 지금 여기
Phase 1         표준 확정 (공통 API·컨테이너·소스규약 문서화 + 담당님 승인)
Phase 2         파일럿 (1~2개 그룹을 표준대로 온보딩 = 레퍼런스 케이스)
Phase 3         셀프서비스 (GitHub push→자동 서비스 무인화)
Phase 4         확산 (전 그룹 온보딩 + 그룹 담당자 체계 정착)
```
